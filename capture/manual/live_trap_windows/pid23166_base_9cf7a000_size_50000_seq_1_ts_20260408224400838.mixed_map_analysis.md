- dump: `pid23166_base_9cf7a000_size_50000_seq_1_ts_20260408224400838.bin`

- sha256: `ce5b171f682bb1f26956749242c7ff9d89b017d7d3d873608e45e845da4ab395`

Goal:

## Result

The zero-island PCs in this run are **not** reached by direct static `b/bl` branches.

## Why the zero-island pages keep appearing in the relay log

There are two layers:

1. **Control-flow feeder**
   - the OLLVM-style selector stubs above choose a continuation through `x30` or `x17`
   - that continuation sometimes lands in a page that is zero in this exact dump

2. **Trap-policy amplifier**
   - the process uses one global exception handler over the whole `0x50000` corridor
   - only the claimed dynamic thread is diverted
   - every other thread hitting the corridor just goes through the generic fallback:

```js
if (!activatePage(pageBase)) return false;
return true;
```

So once any non-claimed thread chooses a zero-page continuation, the relay still logs and activates that page, producing repeated zero-island execute faults.

## Conclusion

The zero-island edges after the first few translated blocks are being fed by:
- the in-JIT indirect target-selection stubs at:
  - `0x9cf7b8f8 .. 0x9cf7b908`
  - `0x9cf7c008 .. 0x9cf7c030`
  - `0x9cf7d2b8 .. 0x9cf7d30c`
- and then amplified by the global trap fallback for non-claimed threads

So the immediate bad feeder is **not** a direct branch to a zero page and **not** a static artifact of the trap window by itself. It is the dynamic dispatcher path choosing a continuation in a page that is zero in this exact snapshot.
